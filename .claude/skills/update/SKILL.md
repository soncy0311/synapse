# Update

docs/의 폴더 및 파일 구조를 점검하고, 지식이 성장함에 따라 필요한 구조 변경을 수행하는 skill이다.

## 사용법

```
/update
```

## Instructions

### Step 1: 현재 구조 파악

docs/ 전체 파일 목록과 각 노드의 frontmatter(id, type, tags)를 수집한다.

```bash
# 파일 구조 확인
find docs/ -name "*.md" | sort

# 각 파일의 type, tags 확인
for f in docs/**/*.md; do head -15 "$f"; echo "---"; done
```

### Step 2: 구조 분석

다음 기준으로 현재 구조의 적절성을 평가한다:

1. **카테고리 포화**: 하나의 디렉토리에 파일이 과도하게 많은가? (10개 이상이면 서브디렉토리 검토)
2. **새로운 도메인**: 기존 4개 타입(sources, entities, concepts, analyses)에 속하지 않는 새로운 분류가 필요한가?
3. **서브디렉토리 필요성**: 특정 태그나 도메인의 노드가 충분히 모여서 서브디렉토리로 그룹핑할 가치가 있는가?
4. **빈 디렉토리**: 노드가 없는 빈 디렉토리가 있는가?
5. **파일명 일관성**: 파일명이 id와 일치하는가? kebab-case 규칙을 따르는가?

### Step 3: 변경 계획 수립

구조 변경이 필요하다고 판단되면, 변경 계획을 사용자에게 제시하고 승인을 받는다.

**제시할 내용:**
- 현재 구조의 문제점
- 제안하는 변경 사항 (새 디렉토리, 파일 이동 등)
- 변경의 이유

**주의사항:**
- 파일 이동 시 다른 노드의 `links`나 `[[wikilink]]`가 깨지지 않는지 확인한다. (id 기반이므로 경로 변경은 링크에 영향 없음)
- 불필요한 구조 변경은 하지 않는다. 현재 구조가 적절하면 "변경 불필요"로 보고한다.

### Step 4: 변경 실행

사용자 승인 후:

1. 필요한 디렉토리를 생성한다.
2. 파일을 이동한다.
3. 변경된 파일에 대해 인덱스를 갱신한다.
   ```bash
   make index
   ```

### Step 5: 결과 보고

- 변경된 디렉토리/파일 목록
- 현재 docs/ 구조 요약 (디렉토리별 파일 수)
